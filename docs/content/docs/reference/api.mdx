Nimbus exposes a REST API (v0.2) built on Ktor. The API is optional and can be enabled in `config/nimbus.toml` or started at runtime with the `api start` console command.

## Authentication

All endpoints except `/api/health` require a Bearer token:

```http title="Request"
Authorization: Bearer <your-token>
```

Configure the token in `config/nimbus.toml`:

```toml title="config/nimbus.toml"
[api]
enabled = true
bind = "0.0.0.0"
port = 8080
token = "your-secret-token"
```

<Callout title="Auto-generated token" type="info">
If no token is configured in `nimbus.toml`, Nimbus **auto-generates** a random token on each start and logs the first 8 characters to the console. The API is never unauthenticated. Set a permanent token in the config for production use.
</Callout>

### Token Scopes

Nimbus uses two token levels:

| Token | Scope | Recipients |
|---|---|---|
| **Admin token** | Full API access (all endpoints) | Proxy/Bridge plugin, manual API calls |
| **Service token** | Service-level endpoints only (services, permissions, groups, displays, proxy sync, maintenance, metrics, network) | Game servers (SDK, Perms plugin) |

The service token is automatically derived from the admin token using HMAC-SHA256. Game servers receive the restricted service token via the `NIMBUS_API_TOKEN` environment variable, while the Velocity proxy receives the full admin token.

Admin-only endpoints (require the admin token): `/api/reload`, `/api/stress/*`, `/api/files/*`, `/api/config`, `/api/loadbalancer`, `/api/nodes`.

<Callout title="Rate limiting" type="info">
The API enforces **per-IP** rate limits: **120 requests/minute** globally and **5 requests/minute** for stress test endpoints. Exceeding the limit returns HTTP 429 (Too Many Requests).
</Callout>

### Token Management

JWT-based scoped tokens are available as an opt-in feature. Enable with `jwt_enabled = true` in the `[api]` section of `nimbus.toml`. Legacy bearer tokens always work regardless of this setting.

#### POST /api/tokens

Generate a scoped JWT token. **Admin-only.**

**Request Body:**

```json title="Request"
{
  "subject": "my-panel",
  "scopes": ["services:read"],
  "expiresInSeconds": 3600
}
```

**Response:**

```json title="Response"
{
  "token": "eyJhbGciOiJIUzI1NiIs..."
}
```

| Status Code | Description |
|-------------|-------------|
| 200 | Token generated |
| 400 | Invalid scopes or parameters |
| 403 | JWT not enabled or not admin |

#### GET /api/tokens/scopes

List all available permission scopes. **Admin-only.**

**Response:**

```json title="Response"
{
  "scopes": [
    "services:read",
    "services:write",
    "groups:read",
    "groups:write",
    "metrics:read",
    "cluster:read",
    "cluster:write",
    "files:read",
    "files:write",
    "config:read",
    "config:write",
    "audit:read",
    "admin"
  ]
}
```

| Status Code | Description |
|-------------|-------------|
| 200 | Success |

<Callout type="info">
The `admin` scope is a superscope that grants access to all endpoints. Other scopes follow a `resource:action` pattern. JWT tokens are validated on every request and rejected if expired or if the requested endpoint is outside the token's scopes.
</Callout>

## Response Format

All responses use JSON. Error responses follow a consistent format:

```json title="Error Response"
{
  "success": false,
  "message": "Description of what went wrong",
  "error": "ERROR_CODE"
}
```

The `error` field contains a machine-readable error code that clients can switch on instead of parsing messages. For successful responses, the `error` field is `null`.

**Common error codes:**

| Error Code | Description |
|------------|-------------|
| `SERVICE_NOT_FOUND` | The requested service does not exist |
| `SERVICE_NOT_READY` | Service exists but is not in READY state |
| `SERVICE_START_FAILED` | Failed to start service (max instances or JAR unavailable) |
| `GROUP_NOT_FOUND` | The requested group does not exist |
| `GROUP_ALREADY_EXISTS` | A group with that name already exists |
| `GROUP_HAS_RUNNING_INSTANCES` | Cannot delete group with running services |
| `VALIDATION_FAILED` | Request failed input validation |
| `INVALID_INPUT` | Invalid parameter value |
| `NOT_FOUND` | Generic resource not found |
| `FORBIDDEN` | Action not allowed |
| `READ_ONLY` | Scope is read-only |
| `PATH_TRAVERSAL` | Path traversal attempt blocked |
| `INTERNAL_ERROR` | Unexpected server error |
| `STRESS_ALREADY_RUNNING` | A stress test is already active |
| `STRESS_NOT_RUNNING` | No stress test is currently active |
| `CLUSTER_NOT_ENABLED` | Cluster mode is not enabled |
| `LOAD_BALANCER_NOT_ENABLED` | Load balancer is not enabled |
| `PROXY_NOT_AVAILABLE` | No Velocity proxy available |
| `COMMAND_NOT_FOUND` | Command does not exist |
| `COMMAND_NOT_REMOTE` | Command does not support remote execution |

## Services

### GET /api/services

List all services. Supports filtering by group, state, and custom state.

**Query Parameters:**

| Parameter | Type | Description |
|-----------|------|-------------|
| `group` | string | Filter by group name |
| `state` | string | Filter by state: `PREPARING`, `STARTING`, `READY`, `STOPPING`, `STOPPED`, `CRASHED` |
| `customState` | string | Filter by custom state (case-insensitive) |

**Example:**

```bash title="Terminal"
curl -H "Authorization: Bearer $TOKEN" \
  "http://localhost:8080/api/services?group=Lobby&state=READY"
```

**Response:**

```json title="Response"
{
  "services": [
    {
      "name": "Lobby-1",
      "groupName": "Lobby",
      "port": 30001,
      "host": "127.0.0.1",
      "nodeId": "worker-1",
      "state": "READY",
      "customState": null,
      "pid": 48201,
      "playerCount": 12,
      "startedAt": "2025-01-15T10:30:00Z",
      "restartCount": 0,
      "uptime": "2h 15m 30s",
      "isStatic": false,
      "bedrockPort": null,
      "sync": {
        "inFlight": false,
        "lastPushAt": "2025-01-15T12:30:00Z",
        "lastPushBytes": 67108864,
        "lastPushFiles": 122,
        "canonicalSizeBytes": 134217728
      }
    }
  ],
  "total": 1
}
```

**`sync` field:** Only populated for services with `[group.sync] enabled = true` (or equivalent for dedicated). Null for non-sync services. Fields:

| Field | Description |
|-------|-------------|
| `inFlight` | `true` while a push is currently in progress (sync lock held) |
| `lastPushAt` | ISO-8601 timestamp of the last successful push, or `null` if never |
| `lastPushBytes` | Sum of the last-pushed manifest's file sizes |
| `lastPushFiles` | File count in the last-pushed manifest |
| `canonicalSizeBytes` | Current on-disk size of the canonical copy on the controller |

| Status Code | Description |
|-------------|-------------|
| 200 | Success |
| 400 | Invalid state filter value |

### POST /api/services/{'{'}name{'}'}/migrate

Move a sync-enabled service to a different cluster node. Stops on the current node (which triggers a state push to the controller's canonical store), then starts on the target (which pulls the canonical back). For non-sync services, data on the source node is lost.

```json title="Request"
{
  "target": "worker-2"
}
```

Use `"target": "local"` or omit the target entirely to move a service back to the controller.

```json title="Response"
{
  "success": true,
  "message": "Service 'Lobby-1' migrated to node 'worker-2'"
}
```

| Status Code | Description |
|-------------|-------------|
| 200 | Migration complete |
| 400 | Invalid request body |
| 404 | Service not found |
| 500 | Migration failed (target offline, push failed, etc.) |

### POST /api/groups

Create a new server group. Writes a TOML config file and reloads.

**Request Body:**

```json title="Request Body"
{
  "name": "SkyWars",
  "type": "DYNAMIC",
  "template": "skywars",
  "software": "PAPER",
  "version": "1.21.4",
  "memory": "2G",
  "maxPlayers": 16,
  "minInstances": 1,
  "maxInstances": 8,
  "playersPerInstance": 16,
  "scaleThreshold": 0.8,
  "idleTimeout": 300,
  "stopOnEmpty": true,
  "restartOnCrash": true,
  "maxRestarts": 5,
  "jvmArgs": ["-XX:+UseG1GC"]
}
```

All fields except `name` and `template` have defaults. Optional fields:

| Field | Default | Description |
|-------|---------|-------------|
| `type` | `DYNAMIC` | `DYNAMIC` or `STATIC` |
| `software` | `PAPER` | `PAPER`, `PURPUR`, `VELOCITY`, `FORGE`, `NEOFORGE`, `FABRIC`, `CUSTOM` |
| `version` | `1.21.4` | Minecraft version |
| `modloaderVersion` | `""` | Modloader version (Forge/NeoForge/Fabric) |
| `jarName` | `""` | Custom JAR filename |
| `readyPattern` | `""` | Custom regex for ready detection |
| `memory` | `1G` | Memory per instance |
| `maxPlayers` | `50` | Max players per instance |
| `minInstances` | `1` | Minimum running instances |
| `maxInstances` | `4` | Maximum running instances |
| `playersPerInstance` | `40` | Players per instance (for scaling) |
| `scaleThreshold` | `0.8` | Scale up when this % full (0.0-1.0) |
| `idleTimeout` | `0` | Stop idle instances after this many seconds (0 = never) |
| `stopOnEmpty` | `false` | Stop when no players |
| `restartOnCrash` | `true` | Auto-restart on crash |
| `maxRestarts` | `5` | Max restart attempts |
| `jvmArgs` | `["-XX:+UseG1GC"]` | JVM arguments |

**Validation Rules:**
- Names must match `^[a-zA-Z0-9_-]{1,64}$`
- Memory format: `512M` or `2G`
- Version format: `1.21.4` or `1.21.4-pre1`
- `minInstances <= maxInstances`
- `scaleThreshold` between 0.0 and 1.0

| Status Code | Description |
|-------------|-------------|
| 201 | Group created |
| 400 | Validation error |
| 409 | Group already exists |

Served on **plaintext HTTP** on the REST API port (default 8080) — without this the agent would need to trust the controller cert before it can fetch it, creating a chicken-and-egg problem. Gated by the **cluster token** (not the REST API token) because this is the same secret used to authenticate agents. Cert material is public-key, not secret.

```json title="200 Response"
}
```

| Status Code | Description |
|-------------|-------------|
| 200 | Cert material returned |
| 401 | Missing or invalid cluster token |

## Service State Sync

Endpoints used by the agent to pull canonical service state from the controller on start and push it back on graceful stop (the `[group.sync]` / `[dedicated.sync]` feature). All three are served on the **REST API port** (default 8080) and gated by the **cluster token** as a query parameter, matching the template download pattern.

<Callout type="info">
These endpoints are mounted on the same port as the REST API (default 8080), not on the TLS cluster port. Authentication uses the cluster token (passed as a `?token=` query parameter), not the REST API bearer token.
</Callout>

### GET /api/services/{'{'}name{'}'}/state/manifest

Returns a manifest of the controller's canonical copy for [name]. The agent uses this to compute which files to pull or push.

**Query params:** `?token=<cluster-token>`

```json title="200 Response"
{
  "files": {
    "eula.txt": { "sha256": "ee27072e...", "size": 10 },
    "plugins/nimbus-sdk.jar": { "sha256": "87280580...", "size": 73275 },
    "libraries/com/mojang/logging/1.6.11/logging-1.6.11.jar": { "sha256": "5d803eb7...", "size": 20706 }
  }
}
```

Returns an **empty manifest** (not 404) if no canonical state exists yet — the agent treats this as "first start, fall back to template".

| Status Code | Description |
|-------------|-------------|
| 200 | Manifest returned (possibly empty) |
| 401 | Invalid cluster token |

---

### GET /api/services/{'{'}name{'}'}/state/file/{'{'}path...{'}'}

Streams the raw bytes of a single file in the canonical copy. Used by the agent's pull path for each file that differs from its local cache.

**Query params:** `?token=<cluster-token>`

**Response:** `application/octet-stream` — raw file bytes.

| Status Code | Description |
|-------------|-------------|
| 200 | File streamed |
| 400 | Path contains `..` or is invalid |
| 401 | Invalid cluster token |
| 404 | File not found in canonical |

---

### POST /api/services/{'{'}name{'}'}/state/sync

Atomic upload of the service's current state. The agent sends a JSON manifest of its target state plus binary parts for every file that differs from the controller's current manifest. The controller stages the upload, validates hashes, and atomically renames staging → canonical on success.

**Query params:** `?token=<cluster-token>`

**Request body:** `multipart/form-data`
- `manifest` (JSON) — the full [StateManifest] describing what the canonical should look like after sync.
- `file:<relpath>` (binary, one per file to upload) — bytes of each changed file. The relative path is encoded in the part name.

Max body size: 10 GB. Only one sync may be in progress per service at a time — concurrent pushes are rejected with 409.

```json title="200 Response"
{
  "success": true,
  "filesReceived": 12,
  "bytesReceived": 4521678,
  "filesInManifest": 122
}
```

| Status Code | Description |
|-------------|-------------|
| 200 | Sync committed |
| 401 | Invalid cluster token |
| 409 | Another sync for this service is already in progress |
| 500 | Sync failed (hash mismatch, IO error, etc.) — staging rolled back |
