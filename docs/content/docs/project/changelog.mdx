## v0.4.6 (April 9, 2026)

Module loading hotfix — bypasses broken ServiceLoader entirely.

### Fixed
- **Module loading on Java 21**: `ServiceLoader` AND direct `META-INF/services` file reading both failed silently on certain Java 21 VPS configurations. Module classes are now declared via `main_class` in `module.properties` and loaded directly with `Class.forName()` — completely bypasses the broken service discovery mechanism
- **Version compatibility skip was a no-op**: Modules flagged for version incompatibility were still loaded because `continue` only exited the version-check loop, not the actual loading loop. Now properly removes incompatible modules before loading
- **Silent `Error` subclass crashes**: `catch (Exception)` in module load/init/enable missed `ServiceConfigurationError`, `NoClassDefFoundError`, and other `Error` subclasses. Changed to `catch (Throwable)` so these are logged instead of crashing or vanishing

### Changed
- **Module properties**: All built-in modules now declare `main_class` in `module.properties`. Third-party modules can use either `main_class` (preferred) or the legacy `META-INF/services` file as fallback

## v0.4.4 (April 9, 2026)

NeoForge symlink fix and module loading improvements.

### Fixed
- **NeoForge startup on Linux**: `Files.walk()` in `findArgsFile` now uses `FOLLOW_LINKS` so it traverses the symlinked `libraries/` directory correctly. Previously, NeoForge/Forge servers crashed instantly with exit code 1 because the args file was not found through the symlink, falling back to `-jar server.jar` which doesn't work for modded servers
- **Module version skip logic**: Modules skipped due to version incompatibility were still attempted for loading. Fixed by properly removing skipped JARs from the load map before ServiceLoader runs
- **Module loading error handling**: Changed `catch (Exception)` to `catch (Throwable)` in module load/init/enable phases to catch `NoClassDefFoundError` and similar linkage errors that previously crashed silently

### Changed
- **Module loading diagnostics**: Added warnings for JARs missing `module.properties`, JARs with no `NimbusModule` implementation, and failed property reads. Makes debugging broken module JARs much easier

## v0.4.3 (April 9, 2026)

Dashboard proxy and chunked modpack upload.

### Added
- **Server-side proxy**: Hosted dashboard (`https://dashboard.nimbuspowered.org`) can now connect to HTTP controllers without users needing TLS certificates. REST requests are proxied through Next.js API routes, bypassing browser mixed-content restrictions
- **WebSocket SSE bridge**: Server-Sent Events bridge for WebSocket connections (console, live events) through the proxy — transparent to the user, auto-detected
- **Chunked modpack upload**: Large modpack uploads (up to 2 GB) work through the proxy by splitting files into 4 MB chunks. Zero RAM usage on client (`File.slice`), proxy (streamed), and controller (direct disk append). Upload progress percentage shown in UI
- **Chunked upload API**: Three new endpoints on the controller — `POST /api/modpacks/upload/init`, `POST /api/modpacks/upload/chunk`, `POST /api/modpacks/upload/finalize`. Stale uploads auto-cleaned after 1 hour

### Changed
- **Login form simplified**: Only requires IP/hostname instead of full URL. Port defaults to 8080 if not specified. No default values in fields
- **Direct uploads preserved**: Non-proxy connections (localhost, same network) still use single-shot streaming for best performance

## v0.4.2 (April 9, 2026)

CORS configuration hotfix for new installations.

### Fixed
- **CORS default missing**: New installs via SetupWizard now include `allowed_origins = ["dashboard.nimbuspowered.org"]` in the generated `nimbus.toml`. Previously the field was omitted, causing the dashboard to fail with CORS errors on fresh setups
- **Unknown config keys crash**: TOML parser now tolerates unknown keys (e.g. typos like `cors_origins`) instead of refusing to start. Nimbus silently ignores unrecognized fields for forward compatibility
- **Wrong key name in docs**: Dashboard guide referenced `cors_origins` instead of the correct `allowed_origins`, leading users to add an invalid key that crashed Nimbus

### Changed
- **ApiConfig default**: `allowed_origins` defaults to `["dashboard.nimbuspowered.org"]` even when the field is absent from `nimbus.toml`, matching the documented behavior

## v0.4.1 (April 8, 2026)

CurseForge modpack support and server pack ZIP import.

### Added
- **CurseForge modpack support**: Import modpacks from CurseForge via slug (`import curseforge:all-the-mods-10`), URL (`import https://curseforge.com/minecraft/modpacks/all-the-mods-10`), or by uploading the server pack ZIP. Requires an optional `[curseforge] api_key` in `nimbus.toml` for API-based fetching
- **Server pack ZIP import**: Upload pre-built CurseForge server pack ZIPs (e.g. `ServerFiles-6.6.zip`) via CLI (`import /path/to/ServerFiles.zip`) or the dashboard. Auto-detects modloader type, version, and Minecraft version from installer JARs and startup scripts
- **Modpack upload API** (`POST /api/modpacks/upload`): Streamed file upload endpoint for the dashboard. Files are written directly to disk with a 64KB buffer — no memory buffering, no OOM for large packs. Parameters passed as query params
- **Dashboard modpack upload**: New upload mode in the Import Modpack dialog with drag-and-drop file picker. Supports both Modrinth/CurseForge sources and server pack ZIP uploads with group type, memory, and instance configuration

### Changed
- **Import command** now accepts CurseForge URLs, `curseforge:slug` prefix, and `.zip` files alongside existing Modrinth sources
- **Memory input normalization**: Bare numbers in memory prompts are auto-suffixed (e.g. `8` → `8G`, `512` → `512M`) across `import` and `create` commands

### Fixed
- **NeoForge startup on Windows**: `TemplateManager` now uses Windows directory junctions (`mklink /J`) as fallback when symlinks fail (no admin rights required). Previously, the `libraries/` symlink silently failed on Windows without Developer Mode, causing NeoForge servers to crash with `-jar server.jar` instead of `@libraries/.../win_args.txt`
- **NeoForge installation detection**: `hasServerJar()` now recognizes modern NeoForge installations that use `libraries/` with args files instead of a standalone server JAR. Prevents the installer from re-running on every service start

## v0.3.1 (April 7, 2026)

Remote management tooling: manage your cloud from anywhere.

### Added
- **Remote CLI** (`nimbus-cli`): Standalone 11MB JLine3 console that connects to the controller's REST API + WebSocket. Same commands, same tab completion, same ANSI formatting as the local console. Interactive setup wizard on first run, saved connection profiles in `~/.nimbus/cli.json`. Supports `--host`, `--port`, `--token`, `--profile` flags
- **Console API endpoints**: `POST /api/console/complete` for server-side tab completion, `WS /api/console/stream` for multiplexed command execution, live event streaming, and remote screen sessions
- **CLI session tracking**: Every Remote CLI connection logged in `cli_sessions` DB table (V3 migration). Tracks client username, hostname, OS, IP address, GeoIP location (via ip-api.com), connection duration, and command count
- **`sessions` command**: `sessions active` shows currently connected CLI sessions, `sessions history [--limit N]` shows historical sessions with user@host, IP, location, duration, commands, and status
- **CLI connected/disconnected events**: Live events in local console and WebSocket showing Remote CLI connections with client identity, IP, location, and OS (e.g., `◆ CLI jonas@DESKTOP-ABC connected from 85.123.45.67 (Berlin, Germany) [Windows 11 amd64]`)
- **GeoIP location lookup**: Async location resolution via ip-api.com for public IPs, skipped for local/private addresses
- **Install scripts**: `install-cli.sh` (Linux/macOS) and `install-cli.ps1` (Windows) for one-command CLI installation

### Changed
- **All 26+ console commands** refactored to use `CommandOutput` interface as canonical execution path. `execute(args)` now delegates to `execute(args, output)` via `ConsoleOutput()` — zero logic duplication between local and remote execution
- **`CommandDispatcher`** now has `dispatch(input, output: CommandOutput)` overload for remote command execution with typed output streaming
- **Release workflow** updated to build and upload `nimbus-cli-<version>.jar` alongside controller and agent JARs

### Fixed
- JLine native access warning on Java 21+ suppressed via auto-relaunch with `--enable-native-access=ALL-UNNAMED` (same pattern as controller)

## v0.3.0 (April 7, 2026)

Competitive features: closing the gaps with SimpleCloud and CloudNET.

### Added
- **Template stacking**: Groups can now compose multiple templates via `templates = ["base", "overlay"]` in group config. Templates are applied in order — later templates override earlier ones. Backwards compatible with single `template` field. Agent and cluster protocol support included
- **Warm pool / prepared services**: New `PREPARED` service state. `WarmPoolManager` pre-stages services in the background (15s loop). Configured per group via `scaling.warm_pool_size`. When scaling triggers, a prepared service starts in seconds instead of minutes
- **Player module** (`nimbus-module-players`): Centralized player tracking as a controller module. In-memory player map + DB persistence (`player_sessions`, `player_tracking` tables). REST endpoints (`/api/players/online`, `/api/players/history/{uuid}`, `/api/players/info/{uuid}`, `/api/players/stats`). Console command `players list|info|history|stats` with Bridge remote execution support (`/cloud players`)
- **Service deployments**: Automatically copy changed files from a service's working directory back to its template on stop. Configured via `lifecycle.deploy_on_stop` and `lifecycle.deploy_excludes`. Hash-based file comparison to only copy actual changes
- **Remote file management API**: Browse, read, write, and delete files on agent-hosted services via the existing REST API. New cluster protocol messages (`FileListRequest/Response`, `FileReadRequest/Response`, `FileWriteRequest/Response`, `FileDeleteRequest/Response`). `RemoteFileProxy` with `CompletableDeferred`-based request/response (10s timeout). Path traversal protection on agent side
- **Proxy event endpoint**: Generic `POST /api/proxy/events` for the Bridge to report player connect/disconnect/switch events. Core emits events on EventBus, modules subscribe. No module-specific logic in Bridge or Core
- **Module auto-sync**: `ModuleManager.syncEmbeddedModules()` automatically updates installed module JARs from embedded versions on startup. No more manual copying after rebuilds
- **`PlayerServerSwitch` event**: New core event emitted when a player moves between servers. Bridge reports via proxy event endpoint
- **`ServiceDeployed` event**: Emitted after deploy-back completes, includes file count
- **`ServicePrepared` / `WarmPoolReplenished` events**: Warm pool lifecycle events with console and WebSocket support

### Changed
- **`PlayerConnected`/`PlayerDisconnected` events** now include `uuid` field (previously name-only)
- **`TemplateRoutes.resolveGlobalDirs`** now includes Pufferfish, Leaf, and Folia (previously only Paper, Purpur, Velocity)
- **Core `PlayersCommand` removed** — replaced entirely by Player Module's richer command with subcommands
- **Player session tracking removed from Core** — `PlayerSessions` table, `MetricsCollector` player logic, and `V1_Baseline` migration no longer include player-specific code. All player tracking is now exclusively in the Player Module
- **Console**: Player connect/disconnect/switch events suppressed from CLI output (available via `players` command and WebSocket)

### Fixed
- Legacy `player_sessions` table (pre-v0.3.0 schema without UUIDs) cleanly dropped and recreated by Player Module migration
- Player module uses `player_tracking` table name to avoid collision with Permissions module's `player_meta` table
- Bridge `reportPlayerConnect` moved from `onPostLogin` (player has no server yet) to `onServerConnected` for correct timing

## v0.2.1 (April 7, 2026)

Bug fixes, improved console feedback, and config validation hardening.

### Fixed
- **Console command feedback**: `start`, `stop`, and `restart` commands now check return values and report success or failure accurately. `start` shows allocated service name and port, `restart` shows the new service name and port
- **Silent memory parsing**: `NodeConnection.parseMemoryMb()` now logs a warning on invalid memory formats (e.g., `"abc"`, `"10GB"`) instead of silently returning 0, which could cause incorrect node placement decisions in cluster mode

### Improved
- **Config validation on startup**: `controller.max_memory` format (must match `^\d+[MmGg]$`), `bedrock.base_port` range (1-65535), and `database.port` range (1-65535) are now validated at config load time instead of failing later with cryptic errors
- **Scaling debug logging**: The scaling engine now logs at DEBUG level when scale-up or scale-down is skipped due to cooldown or pending services, making it easier to diagnose why scaling isn't triggering
- **Proxy sync error logging**: `ProxySyncManager` now logs full stack traces when TOML config parsing fails (motd.toml, tablist.toml, chat.toml), instead of only the exception message

## v0.2.0 (April 7, 2026)

Security, stability, and production readiness.

### Added
- **Database migration system**: `MigrationManager` with `schema_migrations` tracking table, versioned schema changes, auto-bootstrap for pre-0.2.0 databases. Modules register migrations via `ModuleContext.registerMigrations()`. Core V1 (baseline) + V2 (audit log), Perms V1000, Scaling V2000
- **Audit logging**: `AuditCollector` subscribes to EventBus, batch-writes to `audit_log` table. Actor tracking on `NimbusEvent` (`system`, `console`, `api:admin`, `api:service`). Console command `audit [limit] [--action TYPE] [--actor NAME]`, REST endpoint `GET /api/audit` (admin-only, paginated, filterable). Configurable retention via `[audit]` config section (default 90 days)
- **Environment variable overrides**: `NIMBUS_API_TOKEN`, `NIMBUS_DB_TYPE`, `NIMBUS_DB_HOST`, `NIMBUS_DB_PORT`, `NIMBUS_DB_NAME`, `NIMBUS_DB_USER`, `NIMBUS_DB_PASSWORD`, `NIMBUS_CLUSTER_TOKEN` override TOML config values. Agent supports `NIMBUS_AGENT_TOKEN`, `NIMBUS_AGENT_CONTROLLER`. Logged without revealing values
- **Cluster TLS**: Cluster WebSocket server switched from CIO to Netty with native `sslConnector`. Auto-generates self-signed keystore at `config/cluster.jks` if none configured. Agent default URL changed `ws://` to `wss://` (**breaking**). Configurable trust via `tls_verify`, `truststore_path`
- **Local PID persistence**: `ControllerStateStore` persists service PIDs to `state/services.json` — services survive controller restarts
- **Process adoption**: `ProcessHandle.adopt()` re-attaches to running OS processes by PID with Windows-compatible fallback
- **State reconciliation**: Grace period (`reconciliation_delay`, default 10s) before starting minimum instances, allows agents to report recovered services
- **Remote service recovery**: Reconnecting agents report currently running services with PIDs, ports, and state — controller rebuilds registry before scaling
- **JWT scoped API auth** (opt-in via `jwt_enabled = true`): HMAC-SHA256 signed tokens with 13 permission scopes (`services:read/write`, `groups:read/write`, `metrics:read`, `cluster:read/write`, `files:read/write`, `config:read/write`, `audit:read`, `admin`)
- **Token management endpoints**: `POST /api/tokens` to generate scoped JWT tokens, `GET /api/tokens/scopes` to list available scopes
- **Service-level JWT**: Game servers automatically receive scoped JWT with limited permissions when JWT is enabled
- **Leaf server software**: Paper fork from `api.leafmc.one`, treated as Paper-based
- **`ServiceRecovered` event** with CLI display (`RECOVERED`)
- **`AUTH_FAILED`, `INSUFFICIENT_SCOPE` API error codes**
- **`PortAllocator.reserve()`** for recovered service ports

### Changed
- **Org rename**: `KryonixMC` to `NimbusPowered`, package `dev.nimbuspowered.nimbus`
- **Cluster default protocol**: Agent connections now use `wss://` instead of `ws://` (breaking)
- **Docs migration**: VitePress replaced with Fumadocs (Next.js)
- **Interactive pickers**: Arrow-key selection replacing Y/N prompts throughout console

### Fixed
- Migration bootstrap edge cases when upgrading from older versions
- PostgreSQL port 3306 silent override now throws an error
- **Windows file-lock tolerance**: `TemplateManager` skips locked files instead of crashing
- **mainClass reference**: Updated from `dev.kryonix.nimbus.NimbusKt` to `dev.nimbuspowered.nimbus.NimbusKt`
