## v0.6.0 (April 10, 2026)

Dedicated Services: single-instance, fixed-port servers without templates â€” plus a unified memory-reading pipeline and a Controller Info sheet in the dashboard.

### Added
- **Dedicated Services**: a new first-class concept for single-instance, fixed-port servers. Each dedicated service lives in a managed directory under `paths.dedicated/<name>/` (default `dedicated/`), no template required. Users only pick a name, port, software, version and memory; Nimbus auto-creates the directory and downloads the correct server JAR on first start
- **`config/dedicated/*.toml`**: per-service config files loaded at startup alongside group configs
- **`PathsConfig.dedicated`**: new `paths.dedicated` option in `nimbus.toml` (default: `"dedicated"`)
- **Dedicated REST API**: `GET/POST/PUT/DELETE /api/dedicated`, `POST /api/dedicated/{name}/start|stop|restart`, plus `POST /api/dedicated/modpack/import` and `POST /api/dedicated/modpack/upload` for importing Modrinth/CurseForge modpacks or `.mrpack` / server-pack ZIPs directly into a dedicated service
- **`dedicated` console command**: `dedicated list|create|start|stop|restart|delete|info` with an interactive creation wizard mirroring the `create` command for groups
- **Dashboard â€” Dedicated page**: `/dedicated` with a list view, create dialog (auto-fetches software list and versions from `/api/software` â€” same flow as groups), modpack import dialog with source & upload modes, edit dialog for post-creation changes (port, memory, proxy toggle, restart-on-crash), and inline start/stop/restart/delete actions
- **Dashboard â€” Controller Info sheet**: new slide-out sheet in the sidebar footer showing Nimbus version, memory budget with live usage, update-available banner, and a collapsible changelog per version fetched from `docs/content/docs/project/changelog.mdx` on GitHub `main` (5 minute cache)
- **`GET /api/controller/info`**: returns version, uptime, controller JVM heap, services memory budget (`controller.max_memory`), sum of allocated memory for running services, sum of actual RSS, and update-check result
- **`GET /api/controller/changelog`**: parses the docs changelog into per-version entries
- **`ProcessMemoryReader`**: new utility that reads resident set size (RSS) of service processes by PID via `/proc/<pid>/status` on Linux/WSL and `tasklist /FI "PID eq <pid>"` on native Windows
- **`ServiceMemoryResolver`**: single source of truth for per-service memory stats across REST routes, console commands and the dashboard. Reads RSS for local services, uses agent-pushed value for remote (cluster) services, and budgets a realistic JVM overhead (`-Xmx + max(256 MB, -Xmx Ã— 30%)`) so RSS doesn't appear to overflow the configured heap
- **Dashboard sidebar redesign**: split nav into labeled sections â€” **Overview**, **Infrastructure** (Services, Groups, Dedicated, Nodes), **Operations** (Console, Plugins, Stress Test), **Monitoring** (Audit Log), **Modules**. Uses the existing `scrollbar-thin` custom scrollbar style on the info sheet. Login page uses `banner.svg` instead of the square icon

### Changed
- **Proxy forwarding mod sync is now start-time and edit-time**: `ServiceFactory.syncProxyForwardingMods()` is called on every service prepare (groups and dedicated) and ensures the correct mod for the current software is present while cleaning up stale mods from other modloaders (e.g. Forgeâ†’Fabric). `syncDedicatedProxyForwarding()` additionally patches `neoforwarding-server.toml` / `proxy-compatible-forge-server.toml` / `FabricProxy-Lite.toml` with the Velocity secret
- **NeoForge 1.20.1 â†” 1.20.2 swap**: `ensureForwardingMod()` now checks for the _expected_ mod family for the current version instead of any forwarding mod, and removes the stale mod from the other family before installing the correct one â€” so switching a NeoForge group between pre-1.20.2 (proxy-compatible-forge) and 1.20.2+ (NeoForwarding) does the right thing
- **Dedicated service proxy toggle**: `PUT /api/dedicated/{name}` detects `proxyEnabled` changes on modded services and installs/removes the forwarding mod + patches the config immediately (no restart needed to see it in the directory)
- **SDK `reportHealth(tps)`**: the SDK no longer reports `memoryUsedMb`/`memoryMaxMb` â€” memory is read from `/proc` (or `tasklist`) by the controller itself, so every service (modded, custom, dedicated, proxy) gets accurate RSS stats. The API endpoint still accepts the old shape for backwards compatibility but ignores memory fields
- **Agent `ServiceHeartbeat.memoryUsedMb`**: agents now read RSS via `/proc` on their own host and push it in the cluster heartbeat, so the controller shows correct memory for services on remote nodes too
- **EULA auto-accept for dedicated services**: written into the service directory on creation (skipped for Velocity-software dedicated services)
- **v0.6.0 version bump** in `gradle.properties`

### Fixed
- **Services list showed 0 memory for non-SDK services**: previously only the Velocity proxy (which runs the bridge plugin) had live memory stats. All other services â€” modded Paper/Purpur backends, dedicated services, custom JARs â€” showed `0 / <-Xmx> MB`. Now every service reports real resident memory
- **Info sheet showed controller JVM heap instead of services budget**: the old implementation exposed `Runtime.getRuntime().maxMemory()` which is the controller process's own `-Xmx` (~256 MB), not the memory budget available to services. The sheet now reads `controller.max_memory` from `nimbus.toml`
- **RSS exceeded `-Xmx` display**: JVM processes always use more resident memory than their heap (metaspace, thread stacks, code cache, native buffers). Memory max displayed in the UI now includes a realistic overhead budget so `1150 / 1024 MB` becomes `1150 / 1331 MB` (â‰ˆ86% used)
- **PUT `/api/dedicated/{name}` used request-body `name` instead of URL path**: could cause config inconsistency if a client sent a different name in the body. The URL path is now authoritative, and port conflicts with other dedicated services are now detected on update

## v0.5.3 (April 9, 2026)

Smart modded client routing via mod list matching.

### Added
- **Mod list matching for modded client routing**: The proxy now reads the client's mod list (from the Forge handshake) and routes to the group whose server mods best overlap, instead of blindly sending all modded clients to the least-loaded modded server. Prevents a Forge 1.20.1 client from landing on a NeoForge 1.21.1 ATM10 server
- **ModScanner**: New utility (`nimbus-core/.../template/ModScanner.kt`) scans a template's `mods/` directory on group load, extracting mod IDs from NeoForge/Forge `mods.toml`, Fabric `fabric.mod.json`, and legacy `mcmod.info`. Mod IDs are cached on `ServerGroup` and exposed via `GET /api/groups` as `modIds`
- **Protocol version filtering**: Modded client routing now filters groups by Minecraft protocol version â€” a 1.20.1 Forge client (protocol 763) won't match a 1.21.4 NeoForge group (protocol 769)
- **Connection type compatibility**: `LEGACY_FORGE` clients only match Forge groups; `MODERN_FORGE` clients match both Forge and NeoForge groups
- **Alternate group fallback on kick**: When a modded client is kicked from a modded server, the bridge now tries to find a different modded group at the same protocol version before disconnecting

### Changed
- **`GET /api/groups` response**: Now includes `modIds` field (sorted list of mod IDs scanned from the group's template)
- **GroupManager**: Now accepts `templatesDir` and scans mod IDs for FORGE/NEOFORGE/FABRIC groups during `loadGroups()` and `reloadGroups()`
- **Bridge ConnectionListener**: `moddedServerFinder` now accepts connection type, protocol version, and client mod IDs instead of returning a simple `Optional<RegisteredServer>`

## v0.5.0 (April 9, 2026)

Modded client support: automatic routing and Velocity configuration for Forge/NeoForge/Fabric players.

### Added
- **Modded client routing**: Bridge plugin detects Forge/NeoForge/Fabric clients via Velocity's `player.getModInfo()` API and automatically routes them to the least-loaded server from a modded group instead of the Paper lobby. Prevents the "incompatible mods" kick that NeoForge clients get when connecting to vanilla/Paper servers

- **Modded groups cache**: Bridge fetches group software types from `/api/groups` on startup and refreshes on controller reconnect. No proxy restart needed when modded groups are added

### Fixed
- **`announce-forge` placement**: Correctly placed at TOML root level (not under `[advanced]`) matching Velocity's default config structure
- **Dashboard**: Service console commands now use `send()` instead of raw WebSocket writes
- **Dashboard**: Fixed non-null assertion for group in save dialog
- **Dashboard**: Fixed proxy body forwarding, WebSocket session reuse, and missing API fields
- **ScalingEngineTest**: Added missing `PREPARED` branch in exhaustive `when` expression

## v0.4.7 (April 9, 2026)

Module startup ordering fix.

### Fixed
- **Permissions module crash on fresh installs**: `PermissionManager.init()` queried the `permission_groups` table during module `init()`, but database migrations hadn't run yet. Moved DB-dependent initialization to `enable()` which runs after migrations
- **Module lifecycle ordering**: Split `enableAll()` into `initAll()` (registers migrations, commands, routes) and `enableAll()` (DB-dependent setup). Migrations now run between the two phases, guaranteeing tables exist before modules query them

## v0.4.6 (April 9, 2026)

Module loading hotfix â€” bypasses broken ServiceLoader entirely.

### Fixed
- **Module loading on Java 21**: `ServiceLoader` AND direct `META-INF/services` file reading both failed silently on certain Java 21 VPS configurations. Module classes are now declared via `main_class` in `module.properties` and loaded directly with `Class.forName()` â€” completely bypasses the broken service discovery mechanism
- **Version compatibility skip was a no-op**: Modules flagged for version incompatibility were still loaded because `continue` only exited the version-check loop, not the actual loading loop. Now properly removes incompatible modules before loading
- **Silent `Error` subclass crashes**: `catch (Exception)` in module load/init/enable missed `ServiceConfigurationError`, `NoClassDefFoundError`, and other `Error` subclasses. Changed to `catch (Throwable)` so these are logged instead of crashing or vanishing

### Changed
- **Module properties**: All built-in modules now declare `main_class` in `module.properties`. Third-party modules can use either `main_class` (preferred) or the legacy `META-INF/services` file as fallback

## v0.4.4 (April 9, 2026)

NeoForge symlink fix and module loading improvements.

### Fixed
- **NeoForge startup on Linux**: `Files.walk()` in `findArgsFile` now uses `FOLLOW_LINKS` so it traverses the symlinked `libraries/` directory correctly. Previously, NeoForge/Forge servers crashed instantly with exit code 1 because the args file was not found through the symlink, falling back to `-jar server.jar` which doesn't work for modded servers
- **Module version skip logic**: Modules skipped due to version incompatibility were still attempted for loading. Fixed by properly removing skipped JARs from the load map before ServiceLoader runs
- **Module loading error handling**: Changed `catch (Exception)` to `catch (Throwable)` in module load/init/enable phases to catch `NoClassDefFoundError` and similar linkage errors that previously crashed silently

### Changed
- **Module loading diagnostics**: Added warnings for JARs missing `module.properties`, JARs with no `NimbusModule` implementation, and failed property reads. Makes debugging broken module JARs much easier

## v0.4.3 (April 9, 2026)

Dashboard proxy and chunked modpack upload.

### Added
- **Server-side proxy**: Hosted dashboard (`https://dashboard.nimbuspowered.org`) can now connect to HTTP controllers without users needing TLS certificates. REST requests are proxied through Next.js API routes, bypassing browser mixed-content restrictions
- **WebSocket SSE bridge**: Server-Sent Events bridge for WebSocket connections (console, live events) through the proxy â€” transparent to the user, auto-detected
- **Chunked modpack upload**: Large modpack uploads (up to 2 GB) work through the proxy by splitting files into 4 MB chunks. Zero RAM usage on client (`File.slice`), proxy (streamed), and controller (direct disk append). Upload progress percentage shown in UI
- **Chunked upload API**: Three new endpoints on the controller â€” `POST /api/modpacks/upload/init`, `POST /api/modpacks/upload/chunk`, `POST /api/modpacks/upload/finalize`. Stale uploads auto-cleaned after 1 hour

### Changed
- **Login form simplified**: Only requires IP/hostname instead of full URL. Port defaults to 8080 if not specified. No default values in fields
- **Direct uploads preserved**: Non-proxy connections (localhost, same network) still use single-shot streaming for best performance

Remote management tooling: manage your cloud from anywhere.

### Added

## v0.3.0 (April 7, 2026)

Competitive features: closing the gaps with SimpleCloud and CloudNET.

### Added
- **Template stacking**: Groups can now compose multiple templates via `templates = ["base", "overlay"]` in group config. Templates are applied in order â€” later templates override earlier ones. Backwards compatible with single `template` field. Agent and cluster protocol support included
- **Warm pool / prepared services**: New `PREPARED` service state. `WarmPoolManager` pre-stages services in the background (15s loop). Configured per group via `scaling.warm_pool_size`. When scaling triggers, a prepared service starts in seconds instead of minutes
- **Player module** (`nimbus-module-players`): Centralized player tracking as a controller module. In-memory player map + DB persistence (`player_sessions`, `player_tracking` tables). REST endpoints (`/api/players/online`, `/api/players/history/{uuid}`, `/api/players/info/{uuid}`, `/api/players/stats`). Console command `players list|info|history|stats` with Bridge remote execution support (`/cloud players`)
- **Service deployments**: Automatically copy changed files from a service's working directory back to its template on stop. Configured via `lifecycle.deploy_on_stop` and `lifecycle.deploy_excludes`. Hash-based file comparison to only copy actual changes
- **Remote file management API**: Browse, read, write, and delete files on agent-hosted services via the existing REST API. New cluster protocol messages (`FileListRequest/Response`, `FileReadRequest/Response`, `FileWriteRequest/Response`, `FileDeleteRequest/Response`). `RemoteFileProxy` with `CompletableDeferred`-based request/response (10s timeout). Path traversal protection on agent side
- **Proxy event endpoint**: Generic `POST /api/proxy/events` for the Bridge to report player connect/disconnect/switch events. Core emits events on EventBus, modules subscribe. No module-specific logic in Bridge or Core
- **Module auto-sync**: `ModuleManager.syncEmbeddedModules()` automatically updates installed module JARs from embedded versions on startup. No more manual copying after rebuilds
- **`PlayerServerSwitch` event**: New core event emitted when a player moves between servers. Bridge reports via proxy event endpoint
- **`ServiceDeployed` event**: Emitted after deploy-back completes, includes file count
- **`ServicePrepared` / `WarmPoolReplenished` events**: Warm pool lifecycle events with console and WebSocket support

### Changed
- **`PlayerConnected`/`PlayerDisconnected` events** now include `uuid` field (previously name-only)
- **`TemplateRoutes.resolveGlobalDirs`** now includes Pufferfish, Leaf, and Folia (previously only Paper, Purpur, Velocity)
- **Core `PlayersCommand` removed** â€” replaced entirely by Player Module's richer command with subcommands
- **Player session tracking removed from Core** â€” `PlayerSessions` table, `MetricsCollector` player logic, and `V1_Baseline` migration no longer include player-specific code. All player tracking is now exclusively in the Player Module
- **Console**: Player connect/disconnect/switch events suppressed from CLI output (available via `players` command and WebSocket)

### Fixed
- Legacy `player_sessions` table (pre-v0.3.0 schema without UUIDs) cleanly dropped and recreated by Player Module migration
- Player module uses `player_tracking` table name to avoid collision with Permissions module's `player_meta` table
- Bridge `reportPlayerConnect` moved from `onPostLogin` (player has no server yet) to `onServerConnected` for correct timing

## v0.2.0 (April 7, 2026)

Security, stability, and production readiness.

### Added
- **Database migration system**: `MigrationManager` with `schema_migrations` tracking table, versioned schema changes, auto-bootstrap for pre-0.2.0 databases. Modules register migrations via `ModuleContext.registerMigrations()`. Core V1 (baseline) + V2 (audit log), Perms V1000, Scaling V2000
- **Audit logging**: `AuditCollector` subscribes to EventBus, batch-writes to `audit_log` table. Actor tracking on `NimbusEvent` (`system`, `console`, `api:admin`, `api:service`). Console command `audit [limit] [--action TYPE] [--actor NAME]`, REST endpoint `GET /api/audit` (admin-only, paginated, filterable). Configurable retention via `[audit]` config section (default 90 days)
- **Environment variable overrides**: `NIMBUS_API_TOKEN`, `NIMBUS_DB_TYPE`, `NIMBUS_DB_HOST`, `NIMBUS_DB_PORT`, `NIMBUS_DB_NAME`, `NIMBUS_DB_USER`, `NIMBUS_DB_PASSWORD`, `NIMBUS_CLUSTER_TOKEN` override TOML config values. Agent supports `NIMBUS_AGENT_TOKEN`, `NIMBUS_AGENT_CONTROLLER`. Logged without revealing values
- **Cluster TLS**: Cluster WebSocket server switched from CIO to Netty with native `sslConnector`. Auto-generates self-signed keystore at `config/cluster.jks` if none configured. Agent default URL changed `ws://` to `wss://` (**breaking**). Configurable trust via `tls_verify`, `truststore_path`
- **Local PID persistence**: `ControllerStateStore` persists service PIDs to `state/services.json` â€” services survive controller restarts
- **Process adoption**: `ProcessHandle.adopt()` re-attaches to running OS processes by PID with Windows-compatible fallback
- **State reconciliation**: Grace period (`reconciliation_delay`, default 10s) before starting minimum instances, allows agents to report recovered services
- **Remote service recovery**: Reconnecting agents report currently running services with PIDs, ports, and state â€” controller rebuilds registry before scaling
- **JWT scoped API auth** (opt-in via `jwt_enabled = true`): HMAC-SHA256 signed tokens with 13 permission scopes (`services:read/write`, `groups:read/write`, `metrics:read`, `cluster:read/write`, `files:read/write`, `config:read/write`, `audit:read`, `admin`)
- **Token management endpoints**: `POST /api/tokens` to generate scoped JWT tokens, `GET /api/tokens/scopes` to list available scopes
- **Service-level JWT**: Game servers automatically receive scoped JWT with limited permissions when JWT is enabled
- **Leaf server software**: Paper fork from `api.leafmc.one`, treated as Paper-based
- **`ServiceRecovered` event** with CLI display (`RECOVERED`)
- **`AUTH_FAILED`, `INSUFFICIENT_SCOPE` API error codes**
