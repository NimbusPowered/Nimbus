## v0.3.0 (April 7, 2026)

Competitive features: closing the gaps with SimpleCloud and CloudNET.

### Added
- **Template stacking**: Groups can now compose multiple templates via `templates = ["base", "overlay"]` in group config. Templates are applied in order — later templates override earlier ones. Backwards compatible with single `template` field. Agent and cluster protocol support included
- **Warm pool / prepared services**: New `PREPARED` service state. `WarmPoolManager` pre-stages services in the background (15s loop). Configured per group via `scaling.warm_pool_size`. When scaling triggers, a prepared service starts in seconds instead of minutes
- **Player module** (`nimbus-module-players`): Centralized player tracking as a controller module. In-memory player map + DB persistence (`player_sessions`, `player_tracking` tables). REST endpoints (`/api/players/online`, `/api/players/history/{uuid}`, `/api/players/info/{uuid}`, `/api/players/stats`). Console command `players list|info|history|stats` with Bridge remote execution support (`/cloud players`)
- **Service deployments**: Automatically copy changed files from a service's working directory back to its template on stop. Configured via `lifecycle.deploy_on_stop` and `lifecycle.deploy_excludes`. Hash-based file comparison to only copy actual changes
- **Remote file management API**: Browse, read, write, and delete files on agent-hosted services via the existing REST API. New cluster protocol messages (`FileListRequest/Response`, `FileReadRequest/Response`, `FileWriteRequest/Response`, `FileDeleteRequest/Response`). `RemoteFileProxy` with `CompletableDeferred`-based request/response (10s timeout). Path traversal protection on agent side
- **Proxy event endpoint**: Generic `POST /api/proxy/events` for the Bridge to report player connect/disconnect/switch events. Core emits events on EventBus, modules subscribe. No module-specific logic in Bridge or Core
- **Module auto-sync**: `ModuleManager.syncEmbeddedModules()` automatically updates installed module JARs from embedded versions on startup. No more manual copying after rebuilds
- **`PlayerServerSwitch` event**: New core event emitted when a player moves between servers. Bridge reports via proxy event endpoint
- **`ServiceDeployed` event**: Emitted after deploy-back completes, includes file count
- **`ServicePrepared` / `WarmPoolReplenished` events**: Warm pool lifecycle events with console and WebSocket support

### Changed
- **`PlayerConnected`/`PlayerDisconnected` events** now include `uuid` field (previously name-only)
- **`TemplateRoutes.resolveGlobalDirs`** now includes Pufferfish, Leaf, and Folia (previously only Paper, Purpur, Velocity)
- **Core `PlayersCommand` removed** — replaced entirely by Player Module's richer command with subcommands
- **Player session tracking removed from Core** — `PlayerSessions` table, `MetricsCollector` player logic, and `V1_Baseline` migration no longer include player-specific code. All player tracking is now exclusively in the Player Module
- **Console**: Player connect/disconnect/switch events suppressed from CLI output (available via `players` command and WebSocket)

### Fixed
- Legacy `player_sessions` table (pre-v0.3.0 schema without UUIDs) cleanly dropped and recreated by Player Module migration
- Player module uses `player_tracking` table name to avoid collision with Permissions module's `player_meta` table
- Bridge `reportPlayerConnect` moved from `onPostLogin` (player has no server yet) to `onServerConnected` for correct timing

## v0.2.1 (April 7, 2026)

Bug fixes, improved console feedback, and config validation hardening.

### Fixed
- **Console command feedback**: `start`, `stop`, and `restart` commands now check return values and report success or failure accurately. `start` shows allocated service name and port, `restart` shows the new service name and port
- **Silent memory parsing**: `NodeConnection.parseMemoryMb()` now logs a warning on invalid memory formats (e.g., `"abc"`, `"10GB"`) instead of silently returning 0, which could cause incorrect node placement decisions in cluster mode

### Improved
- **Config validation on startup**: `controller.max_memory` format (must match `^\d+[MmGg]$`), `bedrock.base_port` range (1-65535), and `database.port` range (1-65535) are now validated at config load time instead of failing later with cryptic errors
- **Scaling debug logging**: The scaling engine now logs at DEBUG level when scale-up or scale-down is skipped due to cooldown or pending services, making it easier to diagnose why scaling isn't triggering
- **Proxy sync error logging**: `ProxySyncManager` now logs full stack traces when TOML config parsing fails (motd.toml, tablist.toml, chat.toml), instead of only the exception message

## v0.2.0 (April 7, 2026)

Security, stability, and production readiness.

### Added
- **Database migration system**: `MigrationManager` with `schema_migrations` tracking table, versioned schema changes, auto-bootstrap for pre-0.2.0 databases. Modules register migrations via `ModuleContext.registerMigrations()`. Core V1 (baseline) + V2 (audit log), Perms V1000, Scaling V2000
- **Audit logging**: `AuditCollector` subscribes to EventBus, batch-writes to `audit_log` table. Actor tracking on `NimbusEvent` (`system`, `console`, `api:admin`, `api:service`). Console command `audit [limit] [--action TYPE] [--actor NAME]`, REST endpoint `GET /api/audit` (admin-only, paginated, filterable). Configurable retention via `[audit]` config section (default 90 days)
- **Environment variable overrides**: `NIMBUS_API_TOKEN`, `NIMBUS_DB_TYPE`, `NIMBUS_DB_HOST`, `NIMBUS_DB_PORT`, `NIMBUS_DB_NAME`, `NIMBUS_DB_USER`, `NIMBUS_DB_PASSWORD`, `NIMBUS_CLUSTER_TOKEN` override TOML config values. Agent supports `NIMBUS_AGENT_TOKEN`, `NIMBUS_AGENT_CONTROLLER`. Logged without revealing values
- **Cluster TLS**: Cluster WebSocket server switched from CIO to Netty with native `sslConnector`. Auto-generates self-signed keystore at `config/cluster.jks` if none configured. Agent default URL changed `ws://` to `wss://` (**breaking**). Configurable trust via `tls_verify`, `truststore_path`
- **Local PID persistence**: `ControllerStateStore` persists service PIDs to `state/services.json` — services survive controller restarts
- **Process adoption**: `ProcessHandle.adopt()` re-attaches to running OS processes by PID with Windows-compatible fallback
- **State reconciliation**: Grace period (`reconciliation_delay`, default 10s) before starting minimum instances, allows agents to report recovered services
- **Remote service recovery**: Reconnecting agents report currently running services with PIDs, ports, and state — controller rebuilds registry before scaling
- **JWT scoped API auth** (opt-in via `jwt_enabled = true`): HMAC-SHA256 signed tokens with 13 permission scopes (`services:read/write`, `groups:read/write`, `metrics:read`, `cluster:read/write`, `files:read/write`, `config:read/write`, `audit:read`, `admin`)
- **Token management endpoints**: `POST /api/tokens` to generate scoped JWT tokens, `GET /api/tokens/scopes` to list available scopes
- **Service-level JWT**: Game servers automatically receive scoped JWT with limited permissions when JWT is enabled
- **Leaf server software**: Paper fork from `api.leafmc.one`, treated as Paper-based
- **`ServiceRecovered` event** with CLI display (`RECOVERED`)
- **`AUTH_FAILED`, `INSUFFICIENT_SCOPE` API error codes**
- **`PortAllocator.reserve()`** for recovered service ports

### Changed
- **Org rename**: `KryonixMC` to `NimbusPowered`, package `dev.nimbuspowered.nimbus`
- **Cluster default protocol**: Agent connections now use `wss://` instead of `ws://` (breaking)
- **Docs migration**: VitePress replaced with Fumadocs (Next.js)
- **Interactive pickers**: Arrow-key selection replacing Y/N prompts throughout console

### Fixed
- Migration bootstrap edge cases when upgrading from older versions
- PostgreSQL port 3306 silent override now throws an error
- **Windows file-lock tolerance**: `TemplateManager` skips locked files instead of crashing
- **mainClass reference**: Updated from `dev.kryonix.nimbus.NimbusKt` to `dev.nimbuspowered.nimbus.NimbusKt`

- **`health` console command**: Quick overview of service health across all groups
- **`/api/services/health` endpoint**: Aggregated health summary (total, healthy, degraded, unhealthy counts per group)
- **Bridge health integration**: Velocity plugin reports health status to controller
- **`list` command improvements**: Health column, Unicode rendering fixes

### Changed

Java baseline bump and operational improvements.

### Changed

- **Versioned JARs**: JAR filenames now include version number

### Fixed

- Log encoding crash on non-UTF-8 systems

## v0.1.2 (April 5, 2026)

Smart scaling and module extraction.

### Added
- **Smart Scaling module** (`nimbus-module-scaling`):
  - Time-based schedules (e.g., "scale up BedWars at 14:00 on weekends")
  - Predictive warmup from 7-day rolling player count history
  - Per-group configuration in `config/modules/scaling/*.toml`
- **Phased startup**: Proxy services must reach READY state before any backend services start — prevents players connecting to proxies with no backends
- **Dynamic Bridge commands**: Controller pushes command registry to Velocity plugin at runtime

### Changed
- **Module extraction complete**: All permissions and display logic moved from core into standalone module JARs
- **Plugin search**: Unified Hangar + Modrinth search with multi-select UI and automatic dependency resolution

### Fixed
- Release workflow CRLF handling on Windows

## v0.1.1 (April 4, 2026)

Module system architecture and security hardening.

### Added
- **Module system**: Dynamic loading via ServiceLoader + URLClassLoader
  - `NimbusModule` interface with `init()` / `enable()` / `disable()` lifecycle
  - `ModuleContext` providing EventBus, DatabaseManager, ServiceRegistry, GroupManager access
  - Command registration, route registration, migration registration
  - Dependency resolution with topological sort
  - Version compatibility checks (minimum Nimbus version per module)
  - Embedded modules auto-extracted from shadowJar during SetupWizard
- **`modules` command**: `install`, `uninstall`, `list`, `info` subcommands
- **`plugins` command**: Live search across Hangar + Modrinth with version matching
- **`kick` / `broadcast` commands**: Player management from console
- **Least-players lobby balancing**: New players routed to least-populated lobby
- **Module API endpoints**: REST routes registered by modules
- **Event-driven permission cache refresh**: Permissions update without restart

### Changed
- **Package rename**: `dev.nimbus` to `dev.kryonix.nimbus`
- **ProtocolLib dependency removed**: All NMS interactions replaced with API-only approaches
- **Interactive pickers**: Arrow-key selection for module install, group selection, etc.

### Fixed
- **Zip slip vulnerability** in agent template extraction (security)
- **Scaling deadlock** when all instances of a group crash simultaneously
- **Nametag rendering**: MiniMessage tags now parsed correctly in display names
- **Graceful shutdown**: ScalingEngine properly stops before services are terminated

### Security
- Comprehensive security hardening audit across all input boundaries
- Agent template extraction validates paths against directory traversal
- API input sanitization for command injection prevention

## v0.1.0 (April 1, 2026)

Initial public release. The complete Minecraft cloud system, built from scratch.

### Core
- Service lifecycle management: start, stop, restart, crash detection, auto-restart
- Server groups with TOML configuration (`config/groups/*.toml`)
- Port allocation: 25565+ for proxies, 30000+ for backends
- Service naming: `<GroupName>-<N>` convention
- Static services (persistent) and dynamic services (temporary)
- Graceful shutdown order: game servers, then lobbies, then proxies
- Shutdown confirmation: `shutdown` + `shutdown confirm` within 30 seconds

### Scaling
- Player-count-based auto-scaling with configurable thresholds
- Scale-up cooldown: 30 seconds, scale-down cooldown: 120 seconds
- Global service cap via `controller.max_services`
- Paused during active stress tests

### Server Software
- **Paper-based**: Paper, Purpur, Pufferfish, Leaf, Folia
- **Modded**: Forge, NeoForge, Fabric (+ optional Cardboard for Bukkit compat)
- **Custom**: Any server JAR via direct URL
- Auto-download from official sources
- EULA auto-acceptance for supported software
- Via plugin support: ViaVersion, ViaBackwards, ViaRewind with dependency enforcement

### Proxy
- Velocity auto-management: server list, forwarding secret, port config
- Modern forwarding for 1.13+, legacy BungeeCord forwarding for older versions
- Proxy sync: MOTD, tab list, chat formatting via separate TOML configs
- Maintenance mode: global + per-group with whitelist and custom kick message
- Nimbus Bridge: Velocity plugin with hub commands, cloud integration

### API
- REST API v0.2 with bearer token authentication
- Auto-generated 64-char hex token, timing-safe comparison
- Derived HMAC tokens for service-level access
- Rate limiting: 120 req/min global, 5 req/min for stress endpoints
- CORS with configurable origin whitelist
- WebSocket live events stream (`/api/events`)
- Bidirectional console per service (`/api/services/{name}/console`)
- Endpoints: services, groups, status, players, maintenance, stress, reload, shutdown, load balancer, nodes, metrics, audit

### Console
- JLine3 interactive REPL with tab completion
- 30+ commands: `list`, `start`, `stop`, `restart`, `info`, `groups`, `create`, `delete`, `exec`, `log`, `maintenance`, `stress`, `audit`, `reload`, `shutdown`, and more
- ANSI-colored output via ConsoleFormatter
- Banner display on startup

### Database
- SQLite (default), MySQL/MariaDB, PostgreSQL
- Exposed ORM with versioned migrations (`MigrationManager`)
- Core migrations: V1 (baseline tables), V2 (audit log)
- Module migrations via `ModuleContext.registerMigrations()`
- WAL mode + foreign keys for SQLite
- SSL/TLS for MySQL/PostgreSQL connections

### Multi-Node Cluster
- Agent nodes connect via WebSocket (Ktor Netty with native TLS)
- Self-signed keystore auto-generation
- Configurable trust: `tls_verify`, custom truststore
- Placement strategies: least-services, least-memory, round-robin
- Agent state persistence: `AgentStateStore` with PID tracking and service recovery
- Heartbeat monitoring with configurable interval and timeout
- Node failure handling: affected services marked CRASHED, rescheduled

### Load Balancer
- Async TCP load balancer via Java NIO `AsynchronousServerSocketChannel`
- Strategies: round-robin, least-players
- PROXY protocol v2 for client IP passthrough
- Backend health manager with failure tracking
- Connection draining, idle timeout watchdog
- Per-connection semaphore (configurable max connections)

### Plugins & SDK
- **Nimbus SDK**: Server-side API for backend Minecraft servers (Spigot 1.8.8+, Paper, Folia)
- **Nimbus Perms**: Permission plugin with builtin or LuckPerms provider
- **Nimbus Display**: Server selector signs (1.13+) + NPCs via FancyNpcs (Paper 1.20+)
- **Nimbus Bridge**: Velocity plugin for hub commands and cloud integration
- Cross-version compatibility: `VersionHelper`, `SchedulerCompat`, `TextCompat`
- Auto-deployed to backend servers via module-registered `PluginDeployment`

### Bedrock
- Geyser + Floodgate auto-download from GeyserMC API
- Centralized `key.pem` management
- Separate UDP port allocation per proxy
- Auto-generated Geyser config

### Performance
- Aikar's JVM flags (memory-aware tuning)
- Paper/Purpur server config optimization
- `jvm.optimize = true` by default

### Stress Testing
- `stress start <players> [group] [--ramp <seconds>]`
- Distributes simulated players across backends respecting `max_players`
- Scaling engine paused during active tests
- Proxy MOTD + tab reflect simulated counts
- REST API: start, stop, ramp, status

### Monitoring
- Metrics collection with 30-day auto-prune
- Audit logging with 90-day retention (configurable)
- Actor tracking: `system`, `console`, `api:admin`, `api:service`
- `audit` command + `/api/audit` endpoint
- Event bus: sealed event classes with `NimbusEvent.actor` field

### Deployment
- One-command installers: `curl | bash` (Linux/macOS), `irm | iex` (Windows)
- Separate installer for controller and agent nodes
- Java 21 auto-installed if missing
- Auto-updater: GitHub Releases, semver comparison, backup + swap
- Release workflow: GitHub Actions, manual trigger, draft releases
