## Vision

Nimbus exists to make running a Minecraft network **simple, fast, and self-sufficient**. One JAR, one config folder, no external dependencies beyond Java — whether you're running a 20-player lobby or a 2,000-player network across multiple machines.

The long-term goal: **any operator, from solo hobbyist to large network team, can deploy and scale a Minecraft network in minutes** — with modded support, multi-node clusters, and production-grade reliability out of the box.

Nimbus is not trying to be Kubernetes for Minecraft. It's trying to be the tool you reach for *instead* of Kubernetes — because for most Minecraft networks, the complexity of container orchestration is the problem, not the solution.

---

## Guiding Principles

These shape what gets built and what gets cut:

- **Zero external dependencies** — No database server, no web server, no Docker daemon required to get started. SQLite ships embedded, the API server is built in, everything works from a single JAR
- **Convention over configuration** — Sensible defaults that work for 90% of use cases. Advanced operators can customize everything, but new users shouldn't need to
- **Console-first** — The CLI is the primary interface. The dashboard and REST API are complementary, not required
- **Vanilla to modded** — Paper lobbies and NeoForge modpacks are equally first-class. No feature should assume vanilla-only
- **Operator autonomy** — Self-hosted, no phone-home, no telemetry, no vendor lock-in. Your infrastructure, your data

## Now — Current Focus

What's actively being worked on or is next in the queue. Priorities shift as the project grows — expect this section to evolve between releases rather than line up with a specific version number.

### Operator-Facing Features

The core engine, cluster system, and module framework are feature-complete for most use cases, so the next layer of work is what operators actually reach for day-to-day: moderation tooling, asset distribution, player-visible polish. Recent first-party modules (Punishments, Resource Packs) sit in this category, and more will follow wherever real networks hit a gap.

| Goal | Why it matters |
|------|----------------|
| **Fill in operator workflows** | Things every network needs but nobody should have to glue together from third-party plugins — punishments, resource packs, reports/appeals, chat filtering, staff tooling. Each lands as its own module so operators can opt in |
| **Dashboard parity + depth** | Every console workflow should have a first-class dashboard equivalent. Not "a UI for the API", but the thing operators reach for first |

### Stabilization

Making what's already shipped rock-solid under real workloads. This work competes for attention with new features and generally wins when the two collide.

| Goal | Why it matters |
|------|----------------|
| **Dashboard → BETA** | The web dashboard is currently ALPHA. Stabilize the UX, fix edge cases, and reach a state where operators can rely on it daily |
| **Documentation completeness** | Several guides are thin or missing (advanced cluster topologies, backup strategies, migration from CloudNet/SimpleCloud). Docs need to match the feature set |
| **Real-world deployment feedback** | Run Nimbus on production networks, collect operator pain points, and fix what actually breaks — not what we imagine might break |
| **Error messages & diagnostics** | When something goes wrong, the operator should know *what* and *why* without reading source code. Better error context, actionable suggestions, diagnostic commands |

### Community & Ecosystem

| Goal | Why it matters |
|------|----------------|
| **Module developer experience** | The Module API (`nimbus-module-api`) needs better documentation, examples, and a published Maven artifact so third-party developers can build modules |
| **Migration guides** | Operators coming from CloudNet or SimpleCloud need a clear path. Concept mapping, config translation, step-by-step guides |
| **Community module registry** | A way for operators to discover and share community-built modules beyond the built-in ones |

## Next — Path to 1.0

What needs to be true before Nimbus can call itself stable. These are **outcomes**, not a feature list — the specific implementation may change.

### Stable API Contract

The REST API (currently v0.2) needs a stability guarantee. Operators and tool developers need confidence that their integrations won't break on minor updates.

- Versioned API with deprecation policy
- Published OpenAPI/Swagger spec
- Breaking changes only on major versions

### Operational Confidence

Operators running Nimbus in production need to trust that it handles failure gracefully and recovers without intervention.

- Comprehensive integration test suite covering lifecycle, scaling, cluster failover
- Chaos testing: random service kills, network partitions, disk full scenarios
- Documented backup and disaster recovery procedures
- Upgrade path testing (can you go from 0.8 → 1.0 without data loss?)

### Performance Baselines

Quantified performance so operators can capacity-plan:

- Published benchmarks: startup time, scaling latency, API throughput, memory overhead per service
- Performance regression tests in CI
- Tuning guide for large networks (500+ services, 10+ nodes)

### Polish

The kind of work that separates "it works" from "it's good":

- Consistent error handling across all console commands and API endpoints
- Configuration validation with helpful messages on every field
- Graceful handling of every edge case found in real deployments

## Later — Long-Term Goals

These are directions, not commitments. They'll be pursued if there's real demand and they align with the vision. Ordered roughly by expected impact.

### Docker Runtime

Run services as containers instead of bare JVM processes. Useful for operators who want resource isolation, reproducible environments, or CI/CD-driven server images.

- Optional module — bare JVM remains the default
- Configurable base image per group
- Port mapping, volume mounts, resource limits
- Container logs forwarded to controller console

### Pluggable Template Storage (S3 / SFTP)

Template storage backends beyond the local filesystem. Becomes important for large clusters where copying templates between nodes is a bottleneck.

- S3, MinIO, or compatible object storage
- SFTP for remote file servers
- Hash-based cache invalidation (the foundation already exists in agent template sync)

### Multi-Controller High Availability

Remove the controller as a single point of failure. This is the most architecturally complex item on the roadmap and will only be pursued if large networks need it.

- Leader election for placement decisions
- Shared state via database (already the case for most data)
- Automatic failover when the leader dies
- Agents reconnect to any available controller

### Kubernetes Operator

For operators who *do* want Kubernetes — a native operator that manages Nimbus controllers and agents as Kubernetes resources. This is the furthest-out item and may never be built if the Docker runtime module covers enough use cases.

## How This Roadmap Works

This roadmap is organized by **time horizon**, not by version number:

- **Now** — Actively in progress or queued for the next release cycle
- **Next** — Required outcomes before Nimbus reaches 1.0 stability
- **Later** — Long-term directions that depend on community demand

Items move between sections as priorities shift. The [Changelog](/docs/project/changelog) tracks what actually shipped. If you want to influence what comes next, [open an issue](https://github.com/NimbusPowered/Nimbus/issues) or join the discussion.
