## v0.1.x — Foundation

**Theme:** Build a complete, working Minecraft cloud system from scratch.

Everything in v0.1.x was about getting the core right: start servers, manage groups, auto-scale, handle proxies, and provide an API. The full v0.1.x changelog is available on the [Changelog](/docs/project/changelog) page.

### v0.1.0 — Initial Release (April 1, 2026)

The first public release shipped a fully functional cloud system:

- **Core Engine** — Service lifecycle, group configs, port allocation, phased startup (proxies first, then backends)
- **Auto-Scaling** — Player-count-based scaling with configurable thresholds and cooldowns
- **Velocity Proxy** — Auto-managed server list, modern + legacy forwarding, Via plugin support
- **REST API v0.2** — Bearer token auth, WebSocket live events, bidirectional console
- **Console** — JLine3 REPL with 30+ commands, tab completion, ANSI formatting
- **Server Software** — Paper, Purpur, Pufferfish, Folia, Forge, Fabric, NeoForge, Custom
- **SDK + Plugins** — Server SDK, Bridge (Velocity), Permissions, Display (signs + NPCs)
- **Database** — SQLite/MySQL/PostgreSQL via Exposed ORM with versioned migrations
- **Proxy Sync** — MOTD, tab list, chat formatting, maintenance mode
- **TCP Load Balancer** — Async NIO with round-robin/least-players, PROXY protocol v2
- **Bedrock** — Geyser + Floodgate auto-download and config generation
- **Performance** — Aikar's JVM flags, Paper/Purpur config tuning
- **Stress Testing** — Simulated player load with ramping and per-service distribution
- **Multi-Node Cluster** — Agent nodes via WebSocket, TLS with self-signed certs, placement strategies
- **Install Scripts** — One-command installers for Linux/macOS/Windows (controller + agent)
- **Auto-Updater** — GitHub Releases based, semver comparison, backup + swap
- **Cross-Version** — All plugins support Spigot 1.8.8+ through latest Paper/Folia

### v0.1.1 — Module System (April 4, 2026)

- **Dynamic Module Loading** — ServiceLoader + URLClassLoader architecture
- **Module API** — NimbusModule, ModuleContext, lifecycle hooks, dependency resolution
- **Extracted Modules** — Permissions, Display, Smart Scaling as standalone JARs
- **Security Hardening** — Zip slip fix in agent, scaling deadlock fix
- **Plugin Search** — Live search on Hangar + Modrinth with multi-select and dependency resolution
- **Package Rename** — `dev.nimbus` to `dev.kryonix.nimbus` (later `dev.nimbuspowered.nimbus`)

### v0.1.2 — Smart Scaling (April 5, 2026)

- **Smart Scaling Module** — Time-based schedules, predictive warmup from 7-day player history
- **Phased Startup** — Proxy must be READY before any backends start
- **Dynamic Bridge Commands** — Controller pushes command registry to Velocity plugin
- **Module Extraction** — Permissions and Display logic fully moved out of core

### v0.1.3 — Quality of Life (April 5, 2026)

- **Java 16+ Baseline** — Dropped legacy Java 8/11 support
- **`version` Command** — Shows running version, checks for updates
- **JDK Auto-Download** — Multi-provider fallback with streaming progress
- **Versioned JARs** — Smart start scripts with auto-restart after update

### v0.1.4 — API Errors & Health (April 5, 2026)

- **Machine-Readable Errors** — `ApiErrors.kt` with codes like `SERVICE_NOT_FOUND`, `VALIDATION_FAILED`
- **Health System** — `health` command, `/api/services/health` endpoint, Bridge integration
- **Smart Scaling Default** — Module set to `default=false` (opt-in during setup)

## v0.2.x — Security & Production Readiness

**Theme:** Make Nimbus production-ready. Fix the gaps that matter when real players are online and real money is on the line.

### v0.2.0 — Security & Production Readiness (April 7, 2026)

Three pillars: **resilience** (services survive controller crashes), **security** (proper API auth), and **correctness** (no duplicate services after restart).

- **Database Migration System** — `MigrationManager` with versioned schema changes, auto-bootstrap for pre-0.2.0 databases, module migrations via `ModuleContext.registerMigrations()`
- **Audit Logging** — `AuditCollector` with EventBus subscription, actor tracking, `audit` console command, `GET /api/audit` endpoint, configurable retention (default 90 days)
- **Environment Variable Overrides** — `NIMBUS_API_TOKEN`, `NIMBUS_DB_*`, `NIMBUS_CLUSTER_TOKEN`, `NIMBUS_AGENT_TOKEN`, `NIMBUS_AGENT_CONTROLLER` override TOML config
- **Cluster TLS** — Netty engine with native `sslConnector`, auto-generated self-signed keystore, `wss://` default (breaking)
- **Local PID Persistence** — `ControllerStateStore` persists service PIDs to `state/services.json`, re-adopts running processes via `ProcessHandle.adopt()` (Windows-compatible)
- **State Reconciliation** — Grace period (`reconciliation_delay`, default 10s) lets agents report running services before new instances start
- **JWT Scoped API Auth** — Opt-in via `jwt_enabled = true`, HMAC-SHA256 tokens with 13 granular scopes, token management endpoints, service-level JWT
- **Leaf Support** — Leaf (Paper fork) added as supported server software
- **Org Rename** — `KryonixMC` to `NimbusPowered`, package `dev.nimbuspowered.nimbus`
- **Fumadocs** — Documentation migrated from VitePress to Next.js + Fumadocs
- **Interactive Pickers** — Arrow-key selection replacing Y/N prompts throughout

### v0.2.1 — Polish & Hardening (April 7, 2026)

- **Console command feedback** — `start`, `stop`, `restart` now check return values and report actual success/failure. `start` shows allocated service name and port
- **Config validation** — `controller.max_memory`, `bedrock.base_port`, and `database.port` validated at startup instead of failing later
- **Scaling debug logging** — Cooldown and pending-service skips now logged at DEBUG level
- **Memory parsing warnings** — Invalid cluster node memory formats logged instead of silently returning 0
- **Proxy sync error details** — Full stack traces on TOML parse failures

## v0.3.x — Competitive Features

**Theme:** Close the gaps with SimpleCloud and CloudNET. Add the features that make operators choose Nimbus over alternatives.

### v0.3.0 — Competitive Features (April 7, 2026)

Five major features closing the gap with SimpleCloud and CloudNET:

- **Template Stacking** — `templates = ["base", "overlay"]` in group config, applied in order (later overrides earlier). Reduces duplication across groups sharing common plugins. Full agent/cluster protocol support
- **Warm Pool / Prepared Services** — New `PREPARED` service state with `WarmPoolManager` background replenishment (15s loop). Configurable via `scaling.warm_pool_size` per group. Prepared services start in seconds instead of minutes
- **Player Module** (`nimbus-module-players`) — Centralized player tracking as a controller module. In-memory map + DB persistence (`player_sessions`, `player_tracking`). REST endpoints, console commands, Bridge remote execution (`/cloud players`). Clean event-driven architecture via `POST /api/proxy/events`
- **Service Deployments** — Hash-based copy-back of changed files from service to template on stop. Configurable via `lifecycle.deploy_on_stop` and `lifecycle.deploy_excludes`
- **Remote File Management API** — Browse, read, write, and delete files on agent-hosted services via REST. Transparent proxy through cluster WebSocket with path traversal protection
- **Module Auto-Sync** — Embedded module JARs automatically updated on startup, no manual copying after rebuilds


### v0.3.1 — Remote CLI (April 7, 2026)

A standalone binary (`nimbus-cli`, 11MB JAR) that connects to the controller's API and provides the full JLine console experience remotely:

- **Remote CLI** (`nimbus-cli`) — Standalone JLine3 REPL with same commands, tab completion, and ANSI formatting as the local console. Connects via REST (tab completion) + WebSocket (command streaming, live events, screen sessions). Interactive setup wizard on first run, connection profiles saved in `~/.nimbus/cli.json`
- **CommandOutput migration** — All 26+ console commands refactored to use `CommandOutput` interface as canonical execution path. `execute(args)` delegates to `execute(args, output)` via `ConsoleOutput()` — zero logic duplication
- **Console API endpoints** — `POST /api/console/complete` for tab completion, `WS /api/console/stream` for multiplexed command execution, event streaming, and screen sessions
- **CLI session tracking** — Every Remote CLI connection logged in `cli_sessions` DB table with client username, hostname, OS, IP, GeoIP location, duration, and command count. New `sessions` command (`sessions active`, `sessions history`)
- **CLI connected/disconnected events** — Live events in local console showing who connected from where: `◆ CLI jonas@DESKTOP-ABC connected from 85.123.45.67 (Berlin, Germany) [Windows 11 amd64]`
- **GeoIP lookup** — Async location resolution via ip-api.com for public IPs
- **Install scripts** — `install-cli.sh` (Linux/macOS) and `install-cli.ps1` (Windows)

## v0.4.x — Web Dashboard

**Theme:** Manage your cloud from the browser. The dashboard is a separate Next.js application that connects to the existing REST API + WebSocket infrastructure.

<Callout type="warn">
  The Web Dashboard is currently in **ALPHA**. Features may change, and bugs are expected.
</Callout>

### v0.4.0 — Web Dashboard (April 8, 2026)

A self-hosted web UI available at [dashboard.nimbuspowered.org](https://dashboard.nimbuspowered.org):

- **Web Dashboard** (ALPHA) — Full management UI built with Next.js + shadcn/ui. Connect by entering your controller's API URL and token. Features:
  - Service overview with live status, player counts, uptime
  - Group management: view, create, configure server groups
  - Real-time console per service with ANSI rendering
  - Plugin management: search Hangar + Modrinth, install plugins
  - Modpack import from Modrinth, CurseForge, and server pack ZIP upload
  - Stress testing: start, stop, monitor
  - Audit log viewer with filters
  - Node management for multi-node clusters
  - Module configuration (Display, Permissions, Players, SyncProxy)
  - Metrics visualization with charts
  - Responsive design (desktop + mobile)
- **New API endpoints** — Modpack import (`/api/modpacks/*`), plugin management (`/api/plugins/*`), and software versions (`/api/software/*`) added to core for dashboard and external tool consumption

### v0.4.1 — CurseForge & Server Pack Import (April 8, 2026)

CurseForge modpack support and server pack ZIP upload — the most requested modpack feature:

- **CurseForge support** — Import modpacks from CurseForge via slug, URL, or server pack ZIP. Optional `[curseforge] api_key` in `nimbus.toml` for API-based fetching
- **Server pack ZIP import** — Upload pre-built server packs (e.g. ATM10 `ServerFiles-6.6.zip`) via CLI or dashboard. Auto-detects modloader, MC version, and mod count from installer JARs and startup scripts
- **Dashboard upload** — Drag-and-drop file upload in the Import Modpack dialog. Streamed directly to controller (no memory buffering)
- **NeoForge Windows fix** — Directory junctions as symlink fallback, proper `libraries/` detection

### v0.4.2 — CORS Hotfix (April 9, 2026)

- **CORS defaults** — New installs now include `allowed_origins` for the dashboard. Existing installs with missing field get a safe default instead of blocking all dashboard requests

### v0.4.3 — Dashboard Proxy (April 9, 2026)

- **Server-side proxy** — Hosted dashboard can connect to HTTP controllers without TLS certificates. SSE bridge for WebSocket connections through the proxy
- **Chunked modpack upload** — Large modpack uploads (up to 2 GB) split into 4 MB chunks with zero RAM buffering on client, proxy, and controller

### v0.4.4 – v0.4.7 — Module Loading Fixes (April 9, 2026)

A series of rapid-fire fixes for module loading issues discovered on production VPS deployments:

- **NeoForge symlink traversal** — `Files.walk()` now follows symlinks so `libraries/` is found correctly on Linux
- **Module loading bypass** — `ServiceLoader` replaced with `main_class` in `module.properties` for reliable direct class loading on all Java 21 configurations
- **Module lifecycle ordering** — `initAll()` (registers migrations) runs before DB migrations, `enableAll()` (queries DB) runs after — guarantees tables exist before modules query them
- **Error handling** — `catch (Throwable)` instead of `catch (Exception)` to surface `NoClassDefFoundError` and `ServiceConfigurationError`

### How to connect

1. Open [dashboard.nimbuspowered.org](https://dashboard.nimbuspowered.org) in your browser
2. Enter your controller's API URL (e.g., `http://your-server:8080`)
3. Enter your API token (shown in console on startup, or set in `nimbus.toml`)
4. Click **Connect** — the dashboard connects directly to your controller's REST API

<Callout type="info">
  The dashboard runs entirely in your browser. Your API token is stored locally and all requests go directly from your browser to your controller. No data passes through any third-party server.
</Callout>

## v0.5.x — Modded Support

**Theme:** Make modded servers first-class citizens. Automatic routing, proxy configuration, and smart client matching so operators can run multiple modpacks on one network without manual setup.

### v0.5.0 — Modded Client Routing (April 9, 2026)

- **Modded client detection** — Bridge detects Forge/NeoForge clients via Velocity's internal `ConnectionType` (FML handshake marker) and routes them to modded backend groups instead of the Paper lobby
- **Automatic Velocity modded config** — `VelocityConfigGen` detects modded backends and auto-sets `announce-forge = true`, increases `connection-timeout` (10s) and `read-timeout` (60s) for large modpack handshakes
- **Modded groups cache** — Bridge fetches group software types from `/api/groups` on startup and refreshes on controller reconnect

### v0.5.1 — Modded Hotfixes (April 9, 2026)

- **NeoForwarding mod** — NeoForge 1.20.2+ now uses [NeoForwarding](https://modrinth.com/mod/neoforwarding) instead of proxy-compatible-forge (which caused `Empty key` errors)
- **Connection type detection** — Uses Velocity's internal `ConnectionType` instead of `player.getModInfo()` which was empty at initial server selection
- **Kick loop prevention** — Modded clients kicked from a modded server are no longer redirected to a vanilla lobby

### v0.5.2 — Velocity Max Packs (April 9, 2026)

- **`velocity.max-known-packs`** — Auto-sets `-Dvelocity.max-known-packs=512` on proxy JVM args when modded backends exist. Default limit of 64 caused silent disconnects with large modpacks (400+ mods)

### v0.5.3 — Mod List Matching (April 9, 2026)

Smart routing based on mod list overlap — the key feature for networks running multiple modpacks:

- **ModScanner** — Scans template `mods/` directories on group load, extracting mod IDs from NeoForge/Forge `mods.toml`, Fabric `fabric.mod.json`, and legacy `mcmod.info`. Cached on `ServerGroup`, exposed via `GET /api/groups` as `modIds`
- **Protocol version filtering** — Client protocol version must match the group's MC version (prevents 1.20.1 Forge clients landing on 1.21.1 NeoForge servers)
- **Connection type compatibility** — `LEGACY_FORGE` matches Forge only; `MODERN_FORGE` matches Forge or NeoForge
- **Mod overlap scoring** — `|clientMods ∩ serverMods| / |serverMods|` with 0.5 threshold. Best-scoring group wins, ties broken by player count
- **Alternate group fallback on kick** — Modded clients kicked from one group try a different modded group before disconnecting

## Vision — Cloud-Native

**Theme:** Enterprise features for large networks. These are long-term goals, not promises.

### Docker Runtime Module

Run services as Docker containers instead of bare JVM processes:

- Custom service factory registered via module system
- Configurable base image per group
- Port mapping, volume mounts, resource limits
- Container logs forwarded to controller
- Enables CI/CD workflows for custom server images

### S3 / SFTP Template Storage

Pluggable template storage backends beyond local filesystem:

- **S3** — Store templates in AWS S3, MinIO, or compatible storage
- **SFTP** — Remote template storage on dedicated file servers
- Template sync across cluster nodes without manual copying
- Hash-based cache invalidation (already implemented for agent template verification)

### Multi-Controller HA

Remove the single point of failure by supporting multiple controllers:

- Leader election for placement decisions
- Shared state via database (already the case for most data)
- Automatic failover: if leader dies, another controller takes over
- Agent reconnects to any available controller

This is a significant architectural change and will only be pursued if there's real demand from large networks.

### Web Dashboard Enhancements

The ALPHA dashboard will continue to evolve:

- File browser (built on Remote File Management API)
- Template management and editing
- Permission group editor
- Real-time metrics dashboards with historical data
- Multi-language support

### Cloudflare DNS Module

Automatically manage DNS records for services:

- Create SRV records when proxy services start
- Remove records when services stop
- Configurable per group (which domain, which zone)
- Useful for networks with multiple proxy endpoints

**Inspiration:** CloudNET v4's Cloudflare module.

## Competitive Position

How Nimbus compares to the two major Minecraft cloud systems as of April 2026:

### What only Nimbus has

- Built-in REST API (SimpleCloud uses gRPC only, CloudNET has REST as separate module)
- Audit logging with actor tracking
- Stress testing system
- Time-based scaling schedules + predictive warmup
- Performance optimizer (Aikar flags + server config tuning)
- Built-in TCP load balancer with PROXY protocol
- One-command installers
- Live plugin search (Hangar + Modrinth)
- Bedrock/Geyser support
- Folia compatibility across all plugins
- Remote CLI with session tracking and GeoIP (SimpleCloud has CLI but no session auditing)
- Web Dashboard (ALPHA) — browser-based management UI connecting to existing API (SimpleCloud's dashboard is hosted/commercial)
- Smart modded client routing with mod list matching (no manual config, auto-scans template mods)

### What competitors have that we don't (yet)

| Feature | Who has it | Our status |
|---------|-----------|----------|
| Services survive controller crash (local) | SimpleCloud (Screen), CloudNET (Wrapper) | v0.2.0 ✓ |
| State reconciliation on restart | SimpleCloud (auto-reattach) | v0.2.0 ✓ |
| JWT/scoped API auth | CloudNET (module-rest) | v0.2.0 ✓ |
| Warm pool / prepared services | CloudNET (smart module) | v0.3.0 ✓ |
| Cross-server player management | SimpleCloud (Player Droplet) | v0.3.0 ✓ |
| Template stacking | CloudNET (multi-template tasks) | v0.3.0 ✓ |
| Docker runtime | CloudNET (dockerized-services) | Vision |
| S3/SFTP template storage | CloudNET (storage-s3, storage-sftp) | Vision |
| Web dashboard | SimpleCloud (hosted) | v0.4.0 ✓ (ALPHA) |
| BungeeCord support | Both | No plans (Velocity-only by design) |
| Peer-to-peer cluster (no SPOF) | CloudNET | v0.5.0+ (if demand exists) |

### Our philosophy

Nimbus is **lightweight by design**. We don't aim to match every feature of systems that have been developed for years. Instead, we focus on:

1. **Single JAR simplicity** — One file, one command, it works
2. **Smart defaults** — Everything is pre-configured sensibly
3. **Console-first** — No mandatory web UI, no browser required
4. **Built-in essentials** — Permissions, scaling, load balancing, metrics out of the box
5. **Extensible** — Module system for everything else
