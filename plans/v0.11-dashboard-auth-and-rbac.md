# v0.11.0 — Dashboard Auth + Role-Based Access Control

**Status:** Planning
**Target release:** v0.11.0
**Scope:** New `modules/auth` controller module + Bridge command + dashboard auth rewrite + perms-node propagation across all existing modules.

---

## 1. Goals

1. **User-level authentication** on the dashboard — not just machine-level API tokens.
2. **Minecraft-account-verified login** via in-game one-time code **or magic link** (no Microsoft/Mojang OAuth — not allowed for 3rd parties).
3. **Optional TOTP 2FA** per user.
4. **Permission-driven UI** — pages, sidebar entries, and action buttons appear/disable based on permission nodes from the existing Perms Module.
5. **Permission-driven API** — every existing route gets a perm-check; API tokens keep working as admin-superuser.
6. **Zero breaking changes** for existing API-token consumers (CLI, automation, integrations).

### Non-goals (this release)

- Microsoft/Xbox OAuth (not possible for 3rd parties)
- Discord OAuth login (deferred to when Discord-Bridge module lands)
- Password login (deferred; can be added later as convenience, code+TOTP is stronger)
- Password reset flows (N/A — no passwords)

## 2. User Experience

### 2.1 First-time login (Supporter "Max") — Code flow

1. Max has `nimbus.dashboard.access` + `nimbus.dashboard.punishments.*` in-game.
2. Max joins any Nimbus-managed server, types `/dashboard login`.
3. Chat reply: `§a[Nimbus] §fDein Login-Code: §e483 927 §7(60s gültig). Öffne das Dashboard und gib ihn ein.`
4. Max opens `dashboard.nimbuspowered.org` → Login screen has two tabs: **"Minecraft Account"** (default) + **"API Token"** (existing, unchanged).
5. Inside the MC tab, a toggle chooses between **"Login Code"** (default) and **"Magic Link"**. Default = Code.
6. Max types `483927` → session created, skin-head + name appear bottom-left of sidebar.
7. Max sees only the pages his permissions allow: Dashboard overview + Punishments. No Services, no Config, no Backups in sidebar.
8. Inside Punishments page, Max can issue warnings/mutes but the "Ban" and "IP-Ban" buttons are greyed out (missing `nimbus.dashboard.punishments.ban`).

### 2.1b Alternative: Magic Link flow

1. Max is on the dashboard login page, switches to the **"Magic Link"** sub-tab.
2. Max types his username → dashboard shows "Check your in-game chat ✨".
3. In-game, Max receives a clickable Adventure message: `§d✨ §f[Nimbus] §fDein magischer Login-Link ist bereit! §e§l[Klick zum Einloggen]§r §7(60s gültig)`.
4. Clicking opens the dashboard in a browser with `?link=<token>` → auto-consumed, session created, URL cleaned up via `history.replaceState`.
5. If same device: 1 click and he's in. If different device: he can still copy the link (long-press on mobile chat).
6. Rest of the flow identical (TOTP if enabled, sidebar populated by permissions).

**Terminology:** internally the feature is `magic_link` / `MagicLink` (standard industry term, matches Slack/Notion/Vercel/Auth0 conventions — keeps docs googleable). User-facing copy leans into the "magic" theme (`✨`, "magischer Link").

### 2.2 Admin enables 2FA

1. Admin navigates to `Profile → Security`.
2. Clicks "Enable 2FA" → QR code shown (otpauth://totp/...) + 10 recovery codes.
3. Confirms with a TOTP code from authenticator.
4. Next login: code → TOTP prompt → session.

### 2.3 Returning user

- Session-Token in httpOnly cookie, 7-day TTL with rolling refresh on each request.
- Re-auth only needed after 7 days of inactivity, or after explicit logout, or if perms changed significantly (optional).

### 2.4 Session management

- `/dashboard sessions` in-game — list active dashboard sessions (browser fingerprint, IP, last-used).
- `/dashboard logout-all` in-game — revoke all.
- Dashboard `Profile → Sessions` page mirrors the same.

### 4.2 Database schema (migration range 8000+)

```sql
-- V8000
CREATE TABLE dashboard_sessions (
    token_hash   VARCHAR(64) PRIMARY KEY,      -- SHA-256 of the actual token (token itself never stored)
    uuid         VARCHAR(36) NOT NULL,
    name         VARCHAR(16) NOT NULL,         -- snapshot at creation
    created_at   BIGINT NOT NULL,
    expires_at   BIGINT NOT NULL,
    last_used_at BIGINT NOT NULL,

    user_agent   VARCHAR(255),

);
CREATE INDEX idx_sessions_uuid ON dashboard_sessions(uuid);
CREATE INDEX idx_sessions_expires ON dashboard_sessions(expires_at);

-- V8001
CREATE TABLE dashboard_login_challenges (
    challenge_hash VARCHAR(64) PRIMARY KEY,    -- SHA-256 of the code or magic-link token
    kind           VARCHAR(16) NOT NULL,        -- 'code' or 'magic_link'
    uuid           VARCHAR(36) NOT NULL,
    name           VARCHAR(16) NOT NULL,
    created_at     BIGINT NOT NULL,
    expires_at     BIGINT NOT NULL,
    consumed       BOOLEAN DEFAULT FALSE,
    origin_ip      VARCHAR(45)                  -- Only set for magic_link requests (IP that requested link)
);
CREATE INDEX idx_challenges_uuid ON dashboard_login_challenges(uuid);
CREATE INDEX idx_challenges_expires ON dashboard_login_challenges(expires_at);

-- V8002
CREATE TABLE dashboard_totp (
    uuid            VARCHAR(36) PRIMARY KEY,
    secret_enc      VARBINARY(255) NOT NULL,   -- AES-GCM encrypted with server key
    enabled         BOOLEAN DEFAULT FALSE,
    enabled_at      BIGINT
);

-- V8003
CREATE TABLE dashboard_recovery_codes (
    code_hash   VARCHAR(64) PRIMARY KEY,
    uuid        VARCHAR(36) NOT NULL,
    consumed_at BIGINT
);
CREATE INDEX idx_recovery_uuid ON dashboard_recovery_codes(uuid);
```

Token storage: only `sha256(token)` is stored. If the DB leaks, tokens can't be used.

### 4.4 Auth middleware rewrite

In `nimbus-core/src/main/kotlin/.../api/` — extend existing bearer auth:

```kotlin
sealed class AuthPrincipal {
    data object Anonymous : AuthPrincipal()
    data class ApiToken(val token: String) : AuthPrincipal()
    data class UserSession(
        val uuid: UUID,
        val name: String,
        val permissions: PermissionSet,   // snapshot at login, refreshed periodically
        val sessionId: String
    ) : AuthPrincipal()
}

fun AuthPrincipal.hasPermission(node: String): Boolean = when (this) {
    is ApiToken -> true                             // backwards compat
    is UserSession -> permissions.has(node) || permissions.has("nimbus.dashboard.admin")
    Anonymous -> false
}
```

Every route that currently uses `authenticate()` gets an additional `requirePermission("nimbus.dashboard.xxx")` call. Missing perm → 403 with `{ "error": "FORBIDDEN", "required": "nimbus.dashboard.xxx" }`.

### 4.5 Permission resolution

- On successful challenge consumption (code or magic link), `PermissionResolver` asks the Perms Module for all nodes for the UUID (including inherited from groups + tracks).
- Cached on the session row (JSON blob `permissions_snapshot`).
- Refreshed when:
  - Session revalidates after > 5 min (lazy)
  - A `PermissionsChanged(uuid)` event fires (eager, via EventBus subscription)
- Hard refresh: log out + log in.

### 4.6 Bridge / SDK integration

- New command `/dashboard` registered by Auth module's plugin deployment:
  - `/dashboard login` — request 6-digit code (default)
  - `/dashboard login link` — request magic link (alternative trigger from in-game)
  - `/dashboard sessions` — list active sessions
  - `/dashboard logout-all` — revoke all
- Magic link can also be triggered from the dashboard (user types MC name → controller pushes message to in-game player).
- Both on backend servers (via SDK) and proxies (via bridge extension).
- Plugin calls controller via internal cluster-token-authenticated endpoint.
- Magic link Adventure message uses `ClickEvent.OPEN_URL` with the full dashboard URL + `?link=<token>`. URL base comes from `[dashboard] public_url` in `nimbus.toml`.

### 4.7 Config (`config/modules/auth/auth.toml`)

```toml
[sessions]

rolling_refresh = true

[login_challenge]
```

## 8. Security considerations

- **Token storage**: only SHA-256 hash in DB. Raw token lives in httpOnly cookie only.
- **Login challenges**: short-lived (60s), single-use, stored as SHA-256 hash in DB (even the 6-digit code).
  - Numeric 6-digit code is fine because of narrow TTL + rate limit (5 generates/min per UUID) → ~300k attempts in the valid window, attacker gets ~5.
  - Magic-link tokens are 256-bit URL-safe random (`crypto.randomBytes(32).toString('base64url')`).
- **Magic link URL exposure**: tokens appear in URLs, so they carry more leak surface (browser history, referer, clipboard, screen-share). Mitigations:
  - Single-use consumption the moment the page loads
  - `history.replaceState` to strip `?link=` after consumption
  - `Referrer-Policy: no-referrer` header on the login page
  - Magic-link-issued sessions get audited with `login_method=magic_link` for forensic trail
- **Magic-link opt-out**: operators can disable magic link entirely via `magic_link_enabled = false` in auth config, forcing code-only.
- **TOTP secret**: AES-GCM encrypted with a key stored in `config/modules/auth/session.key` (auto-generated, chmod 600).
- **Recovery codes**: 10 codes, SHA-256 hashed, single-use.
- **Rate limiting**: 5 link-code attempts/min per IP, 5 TOTP tries/min per session, global already exists.
- **Session fixation**: new token issued on TOTP success (pre-TOTP token is a "challenge" token, not a full session).
- **CSRF**: cookies are httpOnly + SameSite=Lax; mutating endpoints also accept `Authorization: Bearer` for API clients. Double-submit token pattern for cookie-based dashboard.
- **Audit logging**: every login, logout, 2FA change, session revoke → `AuditCollector`.

## 12. Release notes draft

> **v0.11.0 — Dashboard Authentication & Role-Based Access Control**
>
> The dashboard now supports per-user login tied to Minecraft accounts, with granular permissions powered by the Perms module.
>
> - **Minecraft-account login** — two flavours, both verified via your in-game MC session:
>   - **Login Code**: `/dashboard login` in-game → 6-digit code → enter on dashboard
>   - **Magic Link** ✨: type your username on the dashboard → get a clickable `[Klick zum Einloggen]` message in-game → one click and you're in
>   - No passwords, no Microsoft SSO (not possible for 3rd-party apps).
> - **Optional 2FA** — TOTP via any authenticator app, with recovery codes.
> - **Role-based UI** — sidebar pages and action buttons filter by permission nodes. Supporters see only what they're allowed to touch.
> - **Unified with Perms** — all dashboard nodes live under `nimbus.dashboard.*` and resolve through the existing Perms module with full wildcard support.
> - **Fully backwards compatible** — existing API tokens keep working as admin-superuser for CLI and automation.
